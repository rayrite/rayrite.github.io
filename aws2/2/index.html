🔒 Domain 1 · Secure Architectures
30% of exam weight — the most heavily tested domain. Master IAM, encryption, network security, and the shared responsibility model.
🤝
Shared Responsibility Model
🧠
The Golden Rule AWS is responsible for security OF the cloud (physical, hardware, hypervisor, managed service software). You are responsible for security IN the cloud (your data, IAM, network controls, OS patches on EC2).
▲ HIGHER ABSTRACTION = LESS YOUR RESPONSIBILITY ▲

🟠 AWS Manages

Physical data centers & hardware
Network infrastructure & fiber
Hypervisor (EC2 virtualization layer)
RDS OS & database engine patches
Lambda runtime & OS
S3 storage hardware & durability
DynamoDB infrastructure & scaling
Managed service software updates
Global network backbone

🔵 You Manage

EC2 operating system & patches
Application code & runtime
IAM users, roles, policies
Security Groups & NACLs
Encryption settings on all services
S3 bucket policies & public access
Data classification & sensitivity
VPC design & routing tables
Database users & permissions (in RDS)
⚠️
Classic Exam Trap "Who patches the RDS OS?" → AWS. "Who patches the EC2 OS?" → YOU. The word "managed" in a service name signals AWS takes more responsibility.

Security Groups vs Network ACLs — Most Tested Distinction
FeatureSecurity GroupNetwork ACL (NACL)
Applies toEC2 instance (ENI level)Subnet level
StateStateful — return traffic auto-allowedStateless — must allow both directions
Rule typesAllow rules onlyAllow AND Deny rules
Rule evaluationAll rules evaluated togetherLowest number rule wins first-match
Default (new)Deny all inbound, allow all outboundAllow all inbound and outbound
Blocking IPs❌ Cannot explicitly deny✓ Can block specific IPs
💡
Memory Aid Stateful = "remembers" the conversation. Stateless = forgets instantly, must be re-introduced both ways. If a NACL allows port 443 inbound, you must ALSO explicitly allow ephemeral ports (1024–65535) outbound for the response to get back.

Security Monitoring Decision Tree
GuardDuty
Intelligent threat detection across your AWS account. Analyzes CloudTrail, VPC Flow Logs, DNS logs using ML.
→ "Threat detection"
Inspector
Automated vulnerability scanning for EC2 instances, Lambda, and container images. Software vulnerabilities & network exposure.
→ "Vulnerability scanning"

ML-based sensitive data discovery in S3. Finds PII (names, SSNs, credit cards) and classifies your data automatically.
→ "PII in S3"
WAF
Web Application Firewall. Blocks SQL injection, XSS, and other Layer 7 attacks. Works with CloudFront, ALB, API Gateway.
→ "Web app attacks"
Shield
DDoS protection. Standard = free & automatic. Advanced = paid, 24/7 DDoS response team, financial protection.
→ "DDoS protection"
CloudTrail
Logs every API call in your AWS account. Who did what, when, from where. Essential for compliance & forensics.
→ "Audit / compliance log"

Route 53 Routing Policies
Simple
Returns a single record. No health check magic.
→ Single resource
Weighted
Split traffic by percentage. e.g., 70% → Region A, 30% → Region B.
→ A/B testing, canary
Latency-Based
Routes to the region with lowest latency for the user's location.
→ Performance opt.
Failover
Primary healthy → use it. Primary unhealthy → route to secondary.
→ Active-passive DR
Geolocation
Route based on user's country or continent. Not nearest point.
→ Compliance / locale
Geoproximity
Route based on geographic distance. Adjustable bias to shift traffic.
→ Traffic shifting
Multi-Value
Returns up to 8 healthy records. Client-side load balancing only.
→ Not a real ELB!
⚠️
Geolocation vs Geoproximity Trap Geolocation = user's country/continent (for compliance). Geoproximity = physical distance from your resources (with adjustable bias). These are different!

🧠 Exam Strategy
Multiple-response frameworks, time management, distractor patterns, and a repeatable mental checklist for every question.
📊
Multiple-Response Scoring Reality
🚨
All-or-Nothing No partial credit. You must select EVERY correct answer to earn a point. Selecting A+C when A+D is correct = zero points.
FormatRandom Guess Odds
Single choice (1 of 4)25% (1 in 4)
Select 2 of 510% (1 in 10 combos)
Select 3 of 65% (1 in 20 combos)

This is why elimination is so powerful. Eliminating even ONE wrong answer changes "select 2 of 5" from 10% to 16.7% random odds — a 67% improvement before you apply any knowledge.

⏱️
Time Management — Two-Pass System
Pass 1 — First 90 min: Work through all 65 questions. Answer what you can confidently. For uncertain questions, make your best guess and flag. Target: flag no more than ~13 questions (20%).

Pass 2 — Final 40 min: Return to flagged questions only. Fresh cognitive state + cross-cueing from later questions often unlocks earlier ones.
⏰
Hard Rule Never spend more than 3 minutes on any single question. After 3 minutes, diminishing returns are steep. Flag it, take your best guess, and move on cleanly.
🔍
The 10-Step Mental Checklist for Every Multi-Select Question
  1. Read the stem fully — extract 3 things

    What is the scenario? What is the constraint keyword (most cost-effective / least overhead / highly available)? How many answers to select?

  2. Form a hypothesis before reading options

    What service or pattern would you expect? Anchoring on your own answer prevents the first distractor from hijacking your reasoning.

  3. Evaluate each option independently as TRUE or FALSE

    Grade each option: YES / NO / UNCERTAIN against the stem. Do not consider how options interact yet. This reduces cognitive load.

  4. Eliminate definite NOs immediately

    Anti-cloud patterns, non-existent features, wrong-context services, absolute language ("always," "never") → eliminate with confidence.

  5. If you have exactly the right number of YESes — verify and move on

    Double-check they form a coherent architectural pair, then lock in your answer. Don't second-guess what you know.

  6. If uncertain — anchor on your highest-confidence YES

    Ask: "Given that Option A is correct, which remaining option logically completes this architecture?" Correct answers almost always pair coherently.

  7. Check for mutual exclusivity

    If two options contradict each other, at most one is correct. If one contradicts your anchor, eliminate it. This often resolves uncertainty.

  8. Verify your pair covers ALL stated requirements

    No gaps (missing a requirement), no redundancy (both options do the same thing). Redundant pairs are almost always wrong.

  9. Re-read the constraint keyword one final time

    Does your selected pair actually satisfy "most cost-effective" or "least operational overhead"? This is where most points are lost.

  10. Rate confidence and decide: answer or flag

    High confidence → answer and move on. Medium → consider flagging. Low → flag it, lock in best guess, take a breath, next question.

Common Distractor Patterns — How AWS Writes Wrong Answers
🎯
Real Service, Wrong Context A legitimate AWS service named in a scenario where it doesn't fit. e.g., Snowball Edge as the answer for real-time streaming data ingestion. The service exists, so it feels familiar.
🎯
Correct Service, Wrong Config Right service, wrong detail. e.g., NAT Gateway placed in a private subnet instead of a public subnet. Tests whether you know configuration specifics, not just service names.
🎯
Conflicting Requirements One answer satisfies availability but not cost. Another satisfies cost but not availability. The trap is selecting the one that only satisfies half the question.
🎯
Anti-Cloud Thinking Self-managed where managed is better. Single-AZ where Multi-AZ is stated. Instance store for persistent data. EC2 for a job Lambda can do in <15 minutes.
🎯
Similar Service Confusion SQS vs. SNS. ALB vs. NLB. EBS vs. EFS. CloudFront vs. Global Accelerator. Geolocation vs. Geoproximity. These pairs are the most common distractor strategy on the SAA-C03.
🎯
Absolute Language Words like "always," "never," "all," "only." Cloud architecture involves trade-offs. Absolute claims in options are usually distractors — flag them with skepticism.
