## 1. Executive Summary

The time export file **`T260109.693`** is a **48-byte fixed-width record** flat file generated by the LCD system's **Time Export** module. It contains approved time transactions for organization `693` in **GEMS format**.

This report traces the complete execution path from all possible upstream entry points through the code that physically writes each record to the file. Based on analysis of the file's record structure, the file was generated by the **Oracle `LCD.TimeExport` package** (not the legacy PowerBuilder direct-file path).

Two distinct generations of export logic exist in the codebase:

1. **Modern/Batch Path** (Oracle-side file generation) — **This is the path that produced `T260109.693`**
2. **Legacy PowerBuilder Path** (PB direct `FileOpen`/`FileWrite`) — Generates a different record length (107 bytes)

## 3. Record Format Analysis

The 48-byte record layout matches the Oracle `LCD.TimeExport.p_write_gems` procedure exactly:

| Field          | Position | Length | Format                                | Example      |
| -------------- | -------- | ------ | ------------------------------------- | ------------ |
| `worker_id`    | 1-10     | 10     | Left-padded with spaces               | `11775797  ` |
| `worker_ssn`   | 11-20    | 10     | Left-padded with spaces               | `11775797  ` |
| `actual_date`  | 21-30    | 10     | `MM/DD/YYYY`                          | `03/31/2025` |
| `att_abs_type` | 31-34    | 4      | Left-padded with spaces               | `8000`       |
| `hours`        | 35-44    | 10     | Right-padded with spaces, `999990D99` | `     -9.00` |
| `org_code`     | 45-47    | 3      | Literal org code                      | `693`        |
| `comments`     | 48       | 1      | Single space                          | ` `          |

**Total: 48 bytes**

> **Critical Finding:** The "5 decimal places" apparent in raw text (`-9.00693`) is an optical illusion. The value is actually `     -9.00` (hours, 2 decimal places) immediately concatenated with `693` (org code) and ` ` (comment space), producing `     -9.00693 `. This confirms the file was written by the Oracle `p_write_gems` procedure, not the legacy PB `wf_time_export` function.

### Comparison: Oracle vs. Legacy PB Record Formats

| Path                           | `comments` Field                   | Total Record Length | Evidence in File                |
| ------------------------------ | ---------------------------------- | ------------------- | ------------------------------- |
| **Oracle `p_write_gems`**      | `VARCHAR2(60) := ' '` (1 space)    | **48 bytes**        | ✅ Matches `T260109.693` exactly |
| **Legacy PB `wf_time_export`** | `comments = Space(60)` (60 spaces) | **107 bytes**       | ❌ Does NOT match                |

### 4.5 Main Export Procedure: `lcd.get_time_export`

**File:** `LCD-Oracle/SQL/TIMEXPR3.SQL`

```sql
CREATE OR REPLACE PROCEDURE lcd.get_time_export
  (P_ORG           IN LCD.TIME_EXPORT.ORG_CODE%TYPE,      -- '693'
   FILE_NAME       IN VARCHAR2,                            -- 'T260109.693@TexpSub.693'
   P_WEEK_START_DATE IN VARCHAR2,                         -- e.g. '01/09/2026'
   P_MONDAY_DATE     IN VARCHAR2,                         -- e.g. '01/05/2026'
   comments        OUT VARCHAR2,
   return_value    OUT BOOLEAN)
IS
BEGIN
    -- 1. Establish pipe for PB status communication
    IF LCD.LCDBatch.pipe_establish(P_ORG,'TIME_EXPORT') <> 0 THEN
        RAISE UNABLE_ESTABLISH_PIPE;
    END IF;

    -- 2. Parse FILE_NAME at '@' delimiters
    --    emp_file_name_genesis := SUBSTR(file_name, 1, found_loc1 - 1)
    --    sub_file_name         := SUBSTR(file_name, found_loc1 + 1, found_loc2 - found_loc1 - 1)
    --    emp_file_name_gems    := SUBSTR(file_name, found_loc2 + 1)

    -- 3. Open output files via UTL_FILE
    LCD.Shared.f_open_out_file(P_ORG, lcd.TimeExport.emp_file_name_genesis, ...)
    LCD.Shared.f_open_out_file(P_ORG, lcd.TimeExport.sub_file_name, ...)

    -- 4. Loop through each week_end_date in LCD.TIME_EXPORT
    FOR date_rec IN date_cur(week_ending_day) LOOP
        WEEK_END_DATE   := date_rec.week_end_date;
        WEEK_START_DATE := WEEK_END_DATE - 6;

        -- 5. Generate NoTimeCard records if needed
        NOTIMECARD_FLAG := lcd.TimeExport.do_NoTimeCard(P_ORG, WEEK_START_DATE);

        -- 6. CALL CORE EXPORT ENGINE
        lcd.TimeExport.export_gems(P_ORG, FILE_NAME, TIME_TYPE,
            WEEK_START_DATE, MONDAY_DATE, NOTIMECARD_FLAG,
            COMMENTS, RETURNED);
    END LOOP;

    -- 7. L1-Visa export (if applicable)
    IF lcd.TimeExport.L1_VISA_FLAG AND RETURNED THEN
        lcd.TimeExport.export_L1Visa(...)
    END IF;

    -- 8. Close files, archive, FTP, update event log
END get_time_export;
```

### 4.6 Core Export Engine: `LCD.TimeExport.export_gems`

**File:** `LCD-Oracle/SQL/TIMEXPR2.SQL`

This procedure iterates over workers and their time transactions, calling `p_write_gems` for each record. It also:

- Splits output between **Genesis** (pre-Genesis date) and **GEMS** (post-Genesis date) employee files
- Writes subcontractor records to the sub file
- Generates **NoTimeCard** records for missing employees
- Updates `proc_stat_ind` from `'C'` (created) / `'P'` (posted) / `'R'` (reversed) to exported status

### 4.7 Record Writer: `LCD.TimeExport.p_write_gems`

**File:** `LCD-Oracle/SQL/TIMEXPR2.SQL` (line ~350)

```sql
PROCEDURE p_write_gems(P_OUT_FILE IN UTL_FILE.FILE_TYPE,
                       P_ORG IN VARCHAR2,
                       rec_gems IN RECORD_TYPE_FOR_GEMS,
                       ROWS_WRITTEN IN OUT NUMBER)
IS
    rec_out VARCHAR2(110);
BEGIN
    IF rec_gems.hrs < 0 THEN
        hrs_str := TO_CHAR(rec_gems.hrs,'S999990D99');
    ELSE
        hrs_str := TO_CHAR(rec_gems.hrs,'999990D99');
    END IF;

    hrs_string    := LPAD(hrs_str,10,' ');
    act_date      := TO_CHAR(rec_gems.act_date,'MM/DD/YYYY');
    id_out        := RPAD(rec_gems.wrk_id,10,' ');
    ssn_out       := RPAD(rec_gems.wrk_ssn,10,' ');
    time_type_out := RPAD(rec_gems.time_type,4,' ');

    rec_out := id_out || ssn_out || act_date || time_type_out ||
               hrs_string || P_ORG || comments;
               --  10   +   10    +   10     +      4       +     10     +  3  +   1   = 48

    IF lcd.shared1.f_put_out_line(P_OUT_FILE, rec_out) > 0 THEN
        rows_written := rows_written + 1;
    END IF;
END p_write_gems;
```

## 6. Filename Construction Analysis

### Where does `T260109.693` come from?

The filename is **user-supplied** via the `w_get_exp_names` dialog:

```powerbuilder

// CloseWithReturn(w_get_exp_names, sle_emp_file.text + '@' + sle_sub_file.text)
```

The user **manually changed** the default `TexpEmp.693` to `T260109.693`. The `T` prefix with date encoding (`260109` = Jan 9, 2026) appears to be a **site convention** rather than hardcoded system logic.

The filename then flows through:

1. `w_get_exp_names` → returns `"T260109.693@TexpSub.693"`
2. `cb_export::clicked` → `named = Message.StringParm`
3. `lstr_Job_Schedule.TimeExportFile = named`
4. `w_Job_Schedule` → `ls_File_Named = istr_Job_Schedule.TimeExportFile`
5. `SQLCA.Sub_Job(..., ls_File_Named, ...)`
6. Oracle `LCD.LCDBatch.START_JOB` → `P_FILE_NAME = 'T260109.693@TexpSub.693'`
7. Oracle `lcd.get_time_export` → parses at `@` to get `emp_file_name_genesis = 'T260109.693'`
8. Oracle `LCD.Shared.f_open_out_file` → opens `T260109.693` in the Oracle UTL_FILE directory

## 9. Important Observations

1. **No patched SQL found:** The `tasks/36b/patched/` directory is empty. No patched versions of `TIMEXPR1/2/3.SQL` exist for comparison against the originals in `tasks/36b/original/`.

2. **`ednamed`/`sdnamed` bug:** In `cb_export::clicked` (line ~6926), `ednamed` and `sdnamed` are declared but **never assigned** before being concatenated into `lstr_Job_Schedule.params`. They appear as empty strings in the parameter payload passed to `SQLCA.Sub_Job`. This does not affect the filename (which is in `TimeExportFile`), but may affect downstream logging or FTP control file generation if any code parses the `params` field for file names.

3. **File naming convention:** The `T260109.693` filename follows a **manual user convention** (`T` + `YYMMDD` + `.` + `org`), not a system-generated pattern. The system default is `TexpEmp.693`.

4. **Two generation paths, one active:** The legacy PB `wf_time_export` (direct `FileOpen`/`FileWrite`) is still present in code but **no longer invoked** by the current UI menu structure. All exports now flow through the Oracle batch path.

5. **Record-level evidence:** The 48-byte record length with 1-byte `comments` (trailing space) definitively proves the file was written by Oracle `p_write_gems`, not the PB legacy function.
